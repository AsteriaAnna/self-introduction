几乎每一个刚开始写程序的人，都经历过类似的场景。当我们的TRAE IM无法发送消息的时候，你打开项目，程序还在运行；你刷新页面，没有明显报错；你看控制台，也是一片安静。没有异常信息，没有报错提示，没有任何日志输出。整个程序就像一个完全封闭的黑盒子——**你只能看到结果，却完全不知道中间发生了什么。**

接下来，很多人都会不自觉地进入一种“猜问题”的状态：

- 是不是用户网络不太好？让他刷新试试？
- 会不会是数据库连接慢？要不要重启一下服务？
- 是不是前端按钮状态没更新？还是我刚提交的那行代码有问题

每一个猜测听起来都“有可能”，但真正的问题是：**你没有任何证据。**这种基于猜测的 Debug，不仅效率极低，而且极其不稳定。今天你靠运气猜对了，明天换一个场景，问题依然会重来。在真实的工程环境中，这种排查方式会带来一连串更严重的后果：

- 问题出现时，无法快速定位位置
- 排查过程无法复现，也无法向他人解释
- 团队协作时，每个人都在“各自猜各自的”

这不是经验不足的问题，而是**缺少一种基本能力**。很多非计算机专业的VibeCoder会把注意力全部放在“代码有没有写对”上，却忽略了一件事：**程序本身并不会主动告诉你，它在做什么。**所以我们需要**日志。**

日志（Logging），并不是用来“显得专业”的东西。它的核心作用只有一个：**让程序在运行过程中，把关键信息主动告诉你。**当发现TRAE IM中“消息发不出去”时，一份合理的日志，应该能回答这些问题：

- 请求到底有没有进到这个接口？
- 代码执行到了哪一步？
- 哪个条件分支被走到了？
- 是否出现了异常？发生在什么位置？

日志，就是连接**外部现象**（用户感知到的问题） 与**内部真相**（代码真实执行路径）的唯一桥梁。

1. <br />

## **核心能力一：规范日志输出**

假设一个团队在排查问题时，每个人记录信息的方式都不一样：

- 有人随手 `console.log("error")`
- 有人只打印一句 `"user login"`
- 有人输出一串看不懂的拼接字符串

当这些信息混在一起时，会发生什么？不是“信息多了”，而是**信息无法对齐、无法关联、无法复用**。

这正是没有日志规范的系统在真实工程中最常见的问题。

例如下面这些日志：

```
"Error occurred"
"User logged in"
"Data: 123"
```

它们几乎无法回答任何一个关键问题：

- 错误发生在什么时候？
- 是哪个服务、哪个模块？
- 属于严重问题还是正常流程？
- 是否和某个用户、某次请求有关？

**在日志量很小的时候，你可能还能“猜出来”；一旦进入真实系统规模，这些日志几乎等于不存在。**

为了解决这个问题，工程实践中逐渐形成了一个共识：**日志不应该只是字符串，而应该是结构化数据。**

这就是**结构化日志（Structured Logging）**的核心思想。在现代系统中，最常见、也最成熟的结构化日志格式是 **JSON**。原因很简单：

- 天然是 key–value 结构
- 对人可读
- 对机器极其友好
- 可以被日志平台（如 ELK）直接解析、索引、过滤

一条结构清晰的日志，本质上就是一份**填写完整的“事件记录”**。以下是一条理想的、符合企业级标准的JSON日志结构及其详细字段拆解。这是你未来所有日志记录工作的“黄金模板”。

```
{
  "timestamp": "2026-01-04T10:30:25.123Z",
  "level": "INFO",
  "service_name": "im-message-service",
  "traceId": "a1b2c3d4-e5f6-7890-1234-567890abcdef",
  "thread_name": "http-nio-8080-exec-1",
  "logger_name": "com.vibe.im.service.MessageService",
  "message": "Message saved to database successfully.",
  "details": {
    "userId": "u_12345",
    "groupId": "g_67890",
    "messageId": "msg_abcdef123",
    "db_duration_ms": 52
  },
  "stack_trace": null
}

```

**字段名**

**说明**

**为什么重要？**

**示例**

`timestamp`

日志产生的精确时间，必须使用ISO 8601格式 (`YYYY-MM-DDTHH:mm:ss.SSSZ`)，并统一为UTC时区。

确定事件发生的准确时间线是Debug的基础。UTC时区可以避免因服务器部署在不同地区而导致的时间混乱问题。

`2026-01-04T10:30:25.123Z`

`level`

日志的严重程度，如 \`ERROR\`, \`WARN\`, \`INFO\`, \`DEBUG\`。

`ERROR`：系统异常，必须关注

`WARN`：潜在风险

`INFO`：关键业务流程

`DEBUG`：调试细节（通常不在生产开启）

`INFO`

`service_name`

产生日志的微服务名称。

在微服务体系中，这是**第一层定位**。

当用户说“消息发不出去”，

&#x20;你至少要先知道：是消息服务、用户服务，还是网关出了问题。

`im-message-service`

`traceId`

一个全局唯一的ID，用于串联一次完整请求经过的所有服务和组件。

一次用户操作（如发送消息）就像一个包裹，从前端发出，途径网关、消息服务、用户服务，最终送达数据库。

`a1b2c3d4...`

`thread_name`

执行日志记录代码的线程名称。

在高并发场景下，多个案件同时发生。线程名能帮助我们区分是哪个线程在处理当前这条线索，对于分析并发问题至关重要。

`http-nio-8080-exec-1`

`logger_name`

通常是记录日志的类的全限定名。

你可以不搜索整个项目，而是直接打开对应类。

`com.vibe.im.service.MessageService`

`message`

对日志事件的简洁、人类可读的描述。

一句话说明发生了什么，**不写代码细节，不堆变量值**。

`Message saved successfully.`

`details`

一个可选的JSON对象，用于存放与业务相关的结构化数据。

这里应该放的是：

- userId
- messageId
- orderId
- 耗时
- 状态码

而不是：

- 随意拼接的字符串
- 敏感信息（必须脱敏

`{"userId": "u_12345"}`

`stack_trace`

仅在\`ERROR\`级别日志中出现，记录完整的异常堆栈信息。

**仅用于 ERROR**

用于精确定位异常根源

`java.net.ConnectException...`

在所有字段中，**traceId 是质变点**。在一个复杂的IM系统中，发送一条消息，背后往往会经历：

```
客户端
→ API 网关
→ 消息服务
→ 用户服务（权限校验）
→ 数据库
→ 消息队列
→ 推送服务
→ 客户端
```

如果没有 traceId：

- 你只能靠时间、用户 ID 去“猜”哪些日志属于同一次请求
- 一旦并发上来，几乎不可行

而 traceId 的作用是：**把一次用户操作，在所有服务里的日志“串成一条线”。**它通常在系统入口生成，并通过请求上下文一路传递。每个服务记录日志时，都打印同一个 traceId。当问题出现时，你只需要：搜索 traceId = xxx，你就可以看到这次请求的完整生命周期。

## **核心能力二：何时留下线索？**

学会“怎么写日志”之后，紧接着出现的一个问题是：**日志到底应该打在代码的什么地方？**

这是几乎所有初学者都会卡住的一步。

- 日志打少了，出问题时一片空白
- 日志打多了，自己都不知道该看哪一条

其实，大多数场景下你并不需要复杂的理论，只要记住一个**足够实用的核心原则**：**在“程序的三岔路口”和“关键动作”处留下日志。**你可以把日志当成“路标”：不是一路每隔一米插一根，而是**在决定方向、发生变化的地方做标记**。

1. <br />

### 原则一：在「程序的入口和出口」打日志

这是**最基础、也是最重要**的一类日志。程序的入口，决定“有没有进来”；程序的出口，决定“有没有顺利走完”。

#### ✅ 入口日志（INFO）

- 明确说明：**收到了什么请求**
- 包含：**能唯一定位请求的关键参数**
- **必须脱敏**（绝不打印密码、手机号、身份证）

```
log.info("==> Send message request received. userId={}, groupId={}, type={}",
         request.getUserId(), request.getGroupId(), request.getMessageType());
```

#### ✅ 出口日志（INFO）

- 明确说明：**请求处理结束**
- 必须包含两个信息：
  - **结果（成功 / 失败）**
  - **耗时（ms）**

```
log.info("<== Send message finished. success={}, duration={}ms", success, duration);
```

💡 **耗时日志是新手最容易忽略、但价值极高的数据**它是你以后做性能分析、卡顿排查的第一手证据。

1. <br />

### 原则二：在「关键业务动作或状态变化」时打日志

当程序执行了**会改变系统状态**的操作时，一定要留下清晰记录。这类日志的核心价值是：

- 事后回溯
- 数据审计
- 理解业务流程

#### 推荐写法（INFO）

牢记一个简单公式：

> **谁（Who） + 做了什么（Action） + 对什么（What） + 结果如何（Result）**

```
// 创建群聊
log.info("User [{}] created group [{}].", creatorId, groupId);

// 管理员踢人
log.info("User [{}] removed from group [{}] by admin [{}].",
         targetUserId, groupId, adminId);

// 撤回消息
log.info("Message [{}] recalled by user [{}].", messageId, userId);
```

👉 这种日志**不追求技术细节，而追求“看得懂”**。很多时候，产品、运营、客服也能直接用到这些日志。

1. <br />

## 核心能力三：让 AI 成为你的“日志执行器”

上面两个核心能力，都是让非计算机专业的伙伴，对打日志这件事情有一个初步的认识和规范的了解；那么已经在AI编程的我们，自然也不需要自己动手去一行行的添加日志，我们依然需要扮演好我们的角色。你现在需要思考的就是：**我应该如何描述，才能让AI在这里加上正确的日志**。

很多人写提示词，只写一句：”帮我加点日志“，然而加完日志之后依然定位不到问题，这是失败的根源；一段好的提示词可以让你事半功倍，这件事情，在我们整个课程中我们已经实践过无数次了，在这里我就不做过多的展开赘述了：

**基础 Prompt：为函数补充专业日志**

```
你是一名具有丰富生产环境经验的高级全栈工程师。

【上下文】
- 技术栈：NextJs+SupaBase
- 使用结构化日志（JSON 风格）
- 遵循团队日志规范
- 严禁打印任何敏感信息（如密码、消息内容、手机号等）

【任务】
请为下面的方法补充“可直接用于生产环境”的日志。

【日志要求】
1. 在方法入口和出口处，添加 INFO 级别日志
2. 记录方法整体执行耗时（毫秒）
3. 日志中包含关键业务标识（如 userId、groupId、messageId 等）
4. 对可恢复或被兜底处理的异常，使用 WARN 级别日志
5. 对导致功能失败的异常，使用 ERROR 级别日志，并**必须打印完整异常堆栈**
6. 避免在高频循环中逐条打印日志
7. 日志内容需简洁、明确，便于后续问题排查

【需要加日志的代码】
<请在此粘贴你的方法代码>

```

## 小结

相较于我们前面的小节，本小节内容密度相对比较低；但内容依然很重要，在这一小节中，我们并不是在学习如何手写日志，而是学习如何让AI写出更好更规范，更利于我们定位问题的日志；

通过这一小节你也应该清楚地意识到一件事：AI 并不会“自动生成好日志”。它生成的质量，完全取决于你是否给出了**清晰、结构化、可执行的指令**。当你能够把模糊的需求（“这里加点日志”），转化为明确的规则（入口、出口、耗时、异常、敏感信息），你就已经站在了**专业开发者的工作方式**上。

从现在开始，日志不再是“出了问题才补救”的东西，而是你在设计和编码阶段，就主动构建的一部分可观测性。
